# V1.1 — UI Modernization

**Status:** approved 2026-05-09. Default answers adopted for the *Open Questions* section unless the author redirects during execution.

## Goal

"Windows Terminal but better" — a native, modern Windows-11 feel: dark chrome, tabs in the title bar, plus-button project picker, settings gear next to the system buttons, terminal-typical keyboard and mouse conventions. Functionality is V1-stable; this is a UX-only cycle.

---

## Phase 1 — Custom Window Chrome (foundation)

Foundation for phases 2–4. Without it those phases are ugly workarounds.

- `WindowChrome` (`System.Windows.Shell`) with `CaptionHeight`, `CornerRadius`, `GlassFrameThickness`, `ResizeBorderThickness`. Default frame off, custom layout.
- Dark mode on the title-bar compositing: `DwmSetWindowAttribute(DWMWA_USE_IMMERSIVE_DARK_MODE, true)` via P/Invoke at `Window.SourceInitialized`. Colors the system buttons (min/max/close) in the dark style.
- Custom system buttons (min/max/close) as WPF buttons with `WindowChrome.IsHitTestVisibleInChrome=True` — visually consistent with Edge/Terminal. Hover and pressed states in brand warm tones.
- Drag region: the rest of the caption is a drag surface (default for `WindowChrome`).
- Window border: 1 px in `#332B22`, rounded corners (Win11 style, `CornerRadius="8"`).
- Snap-Layouts (Win11 maximize-hover menu) stay functional — that is `IsHitTestVisibleInChrome` work.

**Acceptance:** window looks like Edge/Terminal (dark, rounded, no light Windows-default header). Min/Max/Close work, Snap-Layouts work, drag works.

---

## Phase 2 — Tabs in Title Bar

- Current `TabControl` out of the body, into the caption region. Caption layout: tabs left (horizontal scroll/overflow on many), system-button cluster right.
- Tab header redesign: more compact, less padding, rounded top corners, active tab visually highlighted.
- State indicator brand visuals stay — tab text foreground continues to mirror `SessionState` (cold/igniting/burning/embers/dead).
- Per-tab close button (`×`). When only one tab exists, hide it (Terminal convention).
- Tabs need `IsHitTestVisibleInChrome=True`, otherwise WindowChrome eats the clicks.
- Tab width: auto down to a min, then horizontal overflow with chevron-scroll like Terminal.

**Out of scope here:** drag-to-reorder, tab-detach, split-pane (all V1.2+).

**Acceptance:** tabs sit in the title bar, active tab clear, tab-close via `×` works, multiple tabs overflow cleanly.

---

## Phase 3 — Plus Button + Project Picker

- `+` button right of the last tab (matched height, brand style, hover).
- Click opens a **project-picker popup** (not a modal): list of all discovered + manual projects, each entry with name + state (running ⇒ greyed/icon) + path as subtext.
- Click an entry → open tab or jump to already-open tab (same behavior as today's sidebar).
- Search field at the top of the popup for quick filter.
- `Ctrl+T` shortcut to open the picker (Terminal-convention).

**Sidebar fate (decision):** the sidebar goes away. The plus-button popup is the single project-discovery surface. Cleaner Terminal-like aesthetic; sidebar was redundant once tabs live in chrome.

**Acceptance:** click `+` → popup with project list → click → tab opens.

---

## Phase 4 — Settings Gear

- Between the tab strip and the system buttons: small gear icon. `IsHitTestVisibleInChrome=True`.
- Click opens the existing M5 `SettingsDialog` — no rewrite.
- Existing "Firepit" menu disappears (gear replaces it).

**Acceptance:** gear visible between tabs and Min, click opens the settings dialog.

## Phase 5 — Clipboard + Terminal-Input UX

Currently broken: `Ctrl+V`, right-click paste. Should behave like Windows Terminal.

- `Ctrl+V` and `Ctrl+Shift+V`: both bound as paste (Terminal convention is `Ctrl+Shift+V`, but `Ctrl+V` is widely expected — accept both).
- **Right-click default:** paste (Linux convention; matches Windows Terminal's default since 2020). Configurable later via settings (`paste-on-right-click` vs `context-menu-on-right-click`).
- `Ctrl+C`: context-aware — if a selection is active, copy; otherwise pass through to the PTY (standard SIGINT behavior). Matches Terminal.
- `Ctrl+Shift+C`: always copy, regardless of selection.
- Implementation: WPF window key bindings catch the shortcuts, dispatch to `WebView2TerminalView.PasteAsync()` / `CopySelectionAsync()`. Bridge protocol gains `{type:'paste',text}` (host→web sends text, web writes via `term.paste(text)`) and `{type:'copy-request'}` (web→host responds with selection text).
- xterm.js: `term.getSelection()` and `term.hasSelection()`.
- Selection style: subtle, brand tone.

**Acceptance:** mouse-select → `Ctrl+C` copies → other app paste yields the same text. Text in clipboard → `Ctrl+V` in Firepit lands in the agent prompt. Right-click pastes without context menu.

## Phase 6 — xterm.js Theming + Scrollbar

- xterm.js scrollbar via CSS (`.xterm-viewport::-webkit-scrollbar`): thin, semi-transparent, brand warm-grey.
- Existing bug "scrollbar drawn behind content" (low-prio per author) gets resolved here as a side effect — a properly styled custom scrollbar layers correctly on its own.
- Theme tokens in settings: `bg`, `fg`, `cursor`, `selection`, ANSI-16 — JSON-editable in `settings.json`. Defaults stay brand warm tones.
- Font loading: currently Cascadia Code (system) — falls back if not installed. **Decision:** embed Cascadia Code as a resource (~700 KB) for deterministic rendering across machines.

**Acceptance:** scrollbar thin, brand-styled, correctly layered. Theme tokens configurable via hand-edit.

## Out of Scope for V1.1

Anything that touches the V2 cockpit stays out:
- Sub-tabs inside a project
- File / markdown / image viewer panes
- Embedded PowerShell pane

V2 hooks from ARCHITECTURE §17 stay unchanged — only chrome and tab-strip style move, the data model does not.

Also out:
- Drag-to-reorder tabs (V1.2)
- Tab-detach to its own window (V1.2)
- Tab-split (two tabs side by side) (V1.2)
- Search / highlight in terminal output (V1.2)
- Quake mode (V2)
- Profile system (V2, via adapter extension)

## Phase Order & Risk

```
Phase 1 (Chrome) ──┬─→ Phase 2 (Tabs in Chrome) ──→ Phase 3 (Plus + Picker)
                   └─→ Phase 4 (Gear)
Phase 5 (Clipboard) ─ independent, can run in parallel or in between
Phase 6 (Scrollbar / theme) ─ independent, best done last
```

Phase 1 is the foundation. Solid `WindowChrome` implementation costs 1–2 hours but pays back across phases 2/3/4.

## Decisions (defaults adopted)

Open questions surfaced during the proposal, with the author's "plan passt" approval default-adopted. Each is reversible during execution if the result feels wrong.

1. **Sidebar:** remove. Plus-button popup is the only project-discovery surface.
2. **Right-click default:** paste (Terminal-convention).
3. **`Ctrl+C` behavior:** context-aware — copies when selection is active, passes through to the PTY otherwise.
4. **Font:** embed Cascadia Code (~700 KB) for deterministic rendering.
5. **Tab close button when single tab:** hide.
