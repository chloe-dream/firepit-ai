# V1.2 — Tab Interactions

**Status:** in progress, ships tonight bundled with V1.1.4 as v0.4.0.
Continuation of V1.1's deferred list per `V1.1-UI-MODERNIZATION.md`
§ *Out of Scope*. Naming convention corrected this iteration: V1.x =
minors with new functionality, V1.x.y = patches on top of a minor
(so V1.11/12/13 retroactively read as V1.1.1/1.1.2/1.1.3).

## Goal

Make the tab strip behave the way a daily-driver Windows-Terminal-class
shell needs to: tabs you can rearrange, sessions you can pop out, and
output you can search. Not the V2 cockpit (that's sub-tabs inside a
project — different abstraction). This is just *more agency over the tab
strip itself*.

## Candidates (deferred from V1.1)

The four items V1.1 explicitly punted to V1.2:

### 1. Drag-to-reorder tabs

WPF `TabControl` does not reorder via drag out of the box. Pattern:
attach `PreviewMouseLeftButtonDown` on `TabItem`, capture, on
`MouseMove` past a hysteresis threshold start a drag-adorner; on
`Drop` on another tab swap indices in the underlying source list.
Order persists into `state.json` so restored sessions reopen in the
user's order, not discovery order.

**Risk:** low. Self-contained, no cross-cutting changes. Drop indicator
needs a brand-warm style — 2 px vertical bar between tabs.

**Estimated scope:** 1 file (MainWindow.xaml.cs interaction handlers),
~150 LOC + a small adorner class.

### 2. Tab-detach to its own window

Pull a tab out of the window into a new top-level window owning the
detached tab's `SessionTab`. The WPF gymnastics: `WebView2` is an
`HwndHost` and re-parenting it across windows is fragile (the Win32
HWND is tied to the host window's airspace). Practical implementations
either tear down + recreate the WebView2 in the new window (loses
scrollback — not great), or keep the original host alive in a hidden
window and proxy the tab content (state-heavy).

**Risk:** medium-high. Architecture decision needed: tear-down-and-
recreate vs hidden-host-proxy. Either way the bridge protocol stays.

**Estimated scope:** new `DetachedWindow.xaml`, IPC channel between
main and detached for tab merge-back, state.json schema bump for
multi-window persistence (optional).

### 3. Tab-split (two tabs side by side)

Split a tab horizontally so two sessions render in the same window.
This collides with the V2 cockpit work (sub-tabs inside a project pane).
If V1.2 ships split, V2's cockpit has to coexist with it cleanly —
otherwise we're refactoring twice.

**Risk:** high. Layout overhaul, conflicts with the V2 cockpit plan in
`SPEC.md` § V2. Decision required up front: does V1.2 split anticipate
V2 cockpit or stay independent?

**Recommendation:** defer to V2 cockpit. V1.2 ships without split.

### 4. Search / highlight in terminal output

xterm.js has `xterm-addon-search` (already-mature). Add the addon, wire
a small search panel (ctrl+F opens, results highlighted in xterm,
next/prev navigation, count display). Embedded resource so it ships
with the bundle.

**Risk:** low. xterm addon does the heavy lifting; we just add UI
chrome and key bindings.

**Estimated scope:** 1 new search panel control, addon integration in
`terminal.html`, ~100 LOC.

## Proposed scoping for V1.2

Ship the **two low-risk items as V1.2** so the minor stays crisp:

- ✅ Drag-to-reorder tabs
- ✅ Search / highlight in terminal output

Defer the others:

- Tab-detach → V1.3 (own minor — needs the architecture decision)
- Tab-split → V2 (folds into cockpit / sub-tabs design)

Rationale: drag-reorder + search are both self-contained and high-impact
for daily use. Detach is heavyweight enough to deserve its own release.
Split competes with V2's cockpit and shouldn't be rushed in.

## Decisions taken

- **Scope cut:** drag-reorder + search ship as V1.2. Detach moved to V1.3
  (own minor — needs the WebView2 re-parenting architecture decision),
  split deferred to V2 (folds into cockpit / sub-tabs design).
- **Drop indicator:** 2 px vertical brand-warm bar (`#F5C97B`) between
  tabs, snaps to left or right half of the hovered tab.
- **Search keybinding:** `Ctrl+Shift+F` (Windows-Terminal convention).
  `Ctrl+F` deliberately stays free for the agent's own readline / fzf
  bindings inside `claude`.
- **Search position:** floating overlay top-right of the active tab.
  Per-tab automatically (each tab has its own xterm.js / addon-search
  instance), so search state never leaks between sessions.

## Implementation summary

**Drag-reorder.** `OnTabPreviewMouseDown` arms a drag from the pressed
TabItem (skipping presses that originate inside the close button); 8 px
hysteresis on `OnTabPreviewMouseMove` triggers `DragDrop.DoDragDrop`.
`Tabs.DragOver` hit-tests the cursor against tab geometry, snaps the
indicator to the nearest gap (left half of hovered tab → insert before,
right half → insert after). `Tabs.Drop` removes + reinserts the source.
Persistence: `OnClosing` now iterates `Tabs.Items` (visual order) rather
than `_openTabs.Values` (insertion order) when writing `state.json`, so
the user's chosen layout survives restart.

**Search.** Added `@xterm/addon-search` 0.15.0 via npm + embedded
resource entry in `Firepit.Web.csproj`. `terminal.html` mounts the
overlay (Cascadia-styled input, match-count badge, prev/next/close
buttons) and wires:
- `Ctrl+Shift+F` opens the overlay and selects the input
- typing live-searches (`searchAddon.findNext` with brand-warm
  decorations: `#5C4D3E` match background, `#F5C97B` active border)
- `Enter` / `Shift+Enter` cycle through matches
- `Esc` closes, clears decorations, returns focus to the terminal
- `onDidChangeResults` drives the count display (`<idx>/<total>`)

The overlay lives inside the WebView2's DOM, so the host stays out of
the search loop entirely — no new bridge messages.

## Acceptance (once scoped)

Drag-to-reorder:
- Drag the second tab onto the first → tabs swap, drop indicator
  visible during drag, dragged tab follows the cursor.
- Order persists across restart when `tabs.persistAcrossRestarts` is on.
- Dragging onto the `+` button area is a no-op (no off-strip drops).

Search:
- `Ctrl+Shift+F` opens an overlay; type → matches highlight in xterm,
  match count visible.
- Enter / arrow keys navigate matches.
- Esc closes the overlay, clears highlights.
- Search is per-tab (each tab has its own search state).

## Out of scope for V1.2

- Tab-detach (V1.3)
- Tab-split (V2 cockpit)
- Multi-window support beyond detach (V2)
- Cross-tab search (V2)
- Search in scrollback older than xterm's buffer limit (xterm-side
  limit, not ours to fix)
