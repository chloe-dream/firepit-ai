# V1.13 — Configurable Font Size + Dynamic UI

**Status:** planned. Continuation of the theming line started in V1.1 Phase 6
(`TerminalThemeSettings`) and V1.11 (toolbar/visual polish).

## Goal

Make the entire Firepit UI scale from a single user-configurable base font size.
Two reasons: a) accessibility (some users want larger text), b) the author has
asked for it and the current hardcoded `FontSize="12"`/`13`/`14` values in XAML
are scattered enough that ad-hoc bumps don't scale evenly.

The UI must reflow cleanly at any font size — no clipped tabs, no fixed heights
that crop the new line height, no overlap between caption and body. The terminal
font size syncs to xterm.js via the existing bridge.

## Scope

**In:**
- New setting `ui.fontSize` in `settings.json` (default `12`). Range clamp `[10, 22]`.
- A single WPF `Resources` token (e.g. `BaseFontSize`) loaded at startup from settings.
  All XAML `FontSize="..."` values become `FontSize="{StaticResource BaseFontSize}"`,
  with a small set of derived sizes (`SmallFontSize` = base - 1, `TitleFontSize` = base + 2)
  for the few places that need them.
- Layout audit: replace every hardcoded `Height="36"` / `Height="32"` (caption,
  toolbar, tab headers) with `Auto` or with a height that scales from base font
  metrics. Specifically: caption row, tab item, toolbar inner padding, settings
  dialog rows.
- Terminal font size: sync `ui.fontSize` (or a separate `terminal.fontSize` if
  decoupling proves useful) into the xterm.js options via the bridge. The bridge
  already has a `theme` message; extend with `font` or piggyback on `theme`.
- Settings dialog: add a font-size input (number stepper or slider) next to the
  projects-root field. Live-preview optional; minimum useful UX is "save → restart
  app picks it up". V1.13 should aim for live-preview if cheap.

**Out:**
- Per-tab font-size overrides (V2).
- High-DPI rework beyond what scaling implies (Windows already handles DPI; we
  ride that).
- Font-family selector — Cascadia Code stays embedded per V1.1 Phase 6.

## Why scattered FontSize values are a problem today

Currently every XAML element sets its own `FontSize="12"` (toolbar buttons,
tab headers, picker rows) or `13` (settings labels, picker name) or `14`
(empty-state text, dialog title). Bumping the base by hand means touching
~20 spots and getting ratios slightly off. A single token + derived sizes
fixes this.

Layout-wise, the immediate offenders that DON'T scale today:
- `MainWindow.xaml`: caption row `Height="36"`, tab item `Height="36"`, settings
  dialog caption `Height="32"`
- `TabToolbar.xaml`: button `Padding="8,5"` (vertical metric)
- `MessageDialog.xaml`: caption `Height="32"`, button row hardcoded margins
- Various `Margin="..."` and `Padding="..."` values that visually keyhole around
  the current font size

The fix is to switch to `Auto` heights where possible and to derive metric
spacing from base-font-size where not (e.g. `CaptionHeight = BaseFontSize * 3`).

## Bridge protocol extension

Per ARCHITECTURE §3.3, the host→web messages currently include `theme`:

```jsonc
{ "type": "theme",  "vars": { "--bg": "...", "--fg": "...", ... } }
```

For V1.13, either:
- **Extend `theme`**: add `font` field — `"font": { "size": 14, "family": "..." }`. Bridge already routes this into xterm.js options.
- **Add `font` message**: `{ "type": "font", "size": 14 }`. Cleaner separation, web side simpler.

Default to the second (separate `font` message). Document in ARCHITECTURE §3.3.

## Acceptance

- Set `ui.fontSize` to 16 in `settings.json` → restart Firepit → caption, tabs,
  toolbar, picker, settings dialog, message dialog all render at the larger size
  with no clipping or overlap.
- Set `ui.fontSize` to 10 → same surfaces render at smaller size, no excess
  whitespace, terminal also scales down.
- The terminal in xterm.js renders at the same size as the rest of the app.
- The settings dialog has a font-size input that updates the value and persists
  to `settings.json`.
- `dotnet test` still passes; no new platform assumptions baked in.

## Risk

The XAML audit is the work. Estimated ~30 spots across the project — mechanical
once the resource token is in place, but each needs visual verification at 10,
12, 16, 22 to catch clipping. Don't merge without that visual matrix.
