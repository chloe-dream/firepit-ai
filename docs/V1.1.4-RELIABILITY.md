# V1.1.4 — Multi-Tab Reliability + Activity Detection + Maximize Polish

**Status:** in progress. Bundles three findings from the 2026-05-10 dogfooding
session, ready for the evening release.

## Goal

Close the rough edges that surfaced during the first day of using Firepit's
own installer. None are showstoppers individually, but they're exactly the
kind of paper-cuts that erode trust in a new tool — the indicator goes idle
when Claude is actually thinking; the 4th tab silently swallows keystrokes;
the selected-tab accent vanishes the moment you maximize. Fix all three,
ship as v0.4.0.

## Scope

**In:**

- **Multi-tab cold-start race.** Parallel `WebView2.EnsureCoreWebView2Async`
  calls under load exceeded the 15s ready-handshake timeout for the 4th tab.
  The tab landed in `Dead` state but the `ready` event still arrived 30s+
  later — leaving a zombie WebView2 with no PTY channel, silently dropping
  every keystroke. Serialize boot via a static `SemaphoreSlim(1, 1)` around
  the critical section.
- **Agent activity during thinking.** `ActivityDetector` only saw byte-stream
  reads, so the indicator demoted Burning → Embers whenever Claude was
  waiting on the model — even though the agent was very much busy. Hook
  Claude Code's `OSC 9 ; 4` tab-progress emissions (state `3` = thinking,
  state `0` = cleared) via `xterm.js`'s `parser.registerOscHandler`, forward
  through the bridge as a `progress` message, pin Burning while active.
- **Maximize clips the top accent.** Custom-chrome WPF windows extend their
  client rect past the work area by `ResizeBorderThickness` on every edge
  when maximized — a known platform quirk. The 2-pixel yellow accent on the
  selected tab sits at the very top of the caption row, so it's the first
  thing to fall off-screen. Compensate by setting a margin on the root grid
  equal to the resize border whenever `WindowState == Maximized`.

**Out:**

- The "stale `WebAssets/<hash>` folders pile up" cleanup — separate concern,
  not user-facing.
- Any changes to the OSC bridge payload format beyond the new `progress`
  message — we keep state and value fields available for future enrichment
  but the host only consumes the boolean for now.
- Reworking the `--continue` resume flow.

## Implementation notes

**WebView2 serialization.** A single static `SemaphoreSlim` in
`WebView2TerminalView` gates `EnsureCoreWebView2Async` through the ready
handshake. Asset extraction and environment lookup stay parallel — they're
idempotent and fast. Trade-off: 4 saved tabs cold-boot in ~30s sequentially
instead of ~10s in parallel. Acceptable because the active tab boots first
and the rest are background work the user doesn't perceive.

**OSC 9;4 parsing.** Lives in `terminal.html`, not in C#. Reasoning: the
host invariant (CLAUDE.md) says "no parsing/interpreting PTY output beyond
timestamping reads." OSC 9;4 is a terminal-protocol message, intended to be
consumed by the terminal — xterm.js is the terminal. Parsing happens at
the right layer; the host only learns "agent says active=true" via a typed
event on `ITerminalView`. `ActivityDetector.NotifyProgress(bool)` pins
Burning until cleared and refreshes `_lastReadAt` so the threshold timer
restarts cleanly when progress flips.

**Maximize compensation.** `OnWindowStateChanged` reads
`WindowChrome.ResizeBorderThickness` and sets the root content's `Margin`
to that thickness when maximized, zero otherwise. Standard pattern from
Windows Terminal / VS Code custom chrome.

## Acceptance

- Cold-start with 4+ saved tabs → all tabs reach Burning, none silently
  drop input. Logs show `WV2 init: waiting for init gate (queued=true)` for
  the queued tabs.
- Ask Claude a long question (e.g. "read 50 files") → tab stays gold/Burning
  the entire time, never demotes to Embers during the thinking pause.
- Maximize the window → the 2px yellow accent on the selected tab is fully
  visible, not clipped. Restore → no margin artifacts, layout returns to
  normal.

## Testing

`ActivityDetector` gets five new unit tests covering pin-past-threshold,
decay-restart-on-clear, lift-from-embers, redundant-signal suppression,
and dead-state ignore. Maximize fix is visual — manual verification on
v0.4.0 install.
