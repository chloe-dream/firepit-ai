# V1.12 — Windows Installer

**Status:** planned 2026-05-09. Pre-requisite for serious dogfooding ("eat our own dogfood"): Firepit must install like a normal Windows app, with shortcuts and a sensible default `projectsRoot`.

## Goal

Replace the current "drop the .exe somewhere and double-click" distribution with a proper Windows installer that:

1. Installs Firepit to a standard location (`%LOCALAPPDATA%\Programs\Firepit\` for per-user; `Program Files\Firepit\` for machine-wide).
2. Creates a Desktop shortcut and a Start-Menu entry. Optional: pin-to-taskbar checkbox.
3. Asks the user during install for a **projects root folder** (default: `%USERPROFILE%\Documents`), and seeds `%APPDATA%\Firepit\settings.json` with that value on first launch.
4. Provides a clean uninstaller (Add/Remove Programs entry, removes app + shortcuts, leaves user data in `%APPDATA%\Firepit` and `%LOCALAPPDATA%\Firepit` untouched by default — checkbox to wipe).
5. Ensures the bundled `firepit-mcp.exe` bridge is resolvable from any Claude Code session Firepit spawns, without requiring user PATH edits or absolute paths in committed `settings.json` files. **Decision (post-#3):** installer adds the install directory to `HKCU\Environment\Path` via an opt-out task ("Add Firepit to PATH"), removes it on uninstall, and triggers `WM_SETTINGCHANGE` so already-running shells / ConPTY children see the update without a logoff.

## Tooling decision (open)

Three candidates, ranked:

1. **Inno Setup** (recommended) — script-driven, free, mature, supports custom UI pages (the "pick projects folder" page). Single `firepit.iss` file checked into the repo. Output: one `FirepitSetup.exe`. Easy CI wiring.
2. **WiX Toolset v5** — produces an MSI (which IT departments prefer), but the XML is heavy and the custom UI page work is more boilerplate. Reserve for if/when corporate distribution becomes a topic.
3. **Velopack** — modern .NET-first installer/auto-updater. Tempting for the auto-update story, but couples us to its conventions and we said "no auto-update in V1". Defer until we want auto-update.

**Default decision**: Inno Setup unless dogfooding immediately surfaces a reason for MSI.

## How the chosen folder reaches Firepit

**Decided: marker file.** Installer writes a single-line text file at
`%APPDATA%\Firepit\first-run-projects-root.txt` containing the chosen path.
Firepit's `JsonSettingsStore.Load()` checks for the marker on every load — if
it exists *and* `settings.json` does not, it writes a full `settings.json`
from `FirepitSettings.Defaults with { ProjectsRoot = <marker contents> }`,
then deletes the marker. Re-installs over an existing settings file are
no-ops on the projects root (the user's choice survives).

Why not a pre-seeded partial JSON: `FirepitSettings` is a positional record
with non-nullable required fields. A partial JSON deserializes to `null` for
missing required strings, which would crash on first read. The marker keeps
schema ownership inside Firepit — installer writes one string, the app
synthesizes the schema-correct default object.

Why not a CLI arg + post-install launch: more moving parts (App.Main parsing,
race with single-instance guard, install-as-system-but-launch-as-user issues).
The marker is dumb and robust.

## Acceptance

- Double-click `FirepitSetup.exe` → wizard runs → projects-folder page appears → after install, Firepit shortcut on Desktop and in Start.
- First launch: project picker reflects the chosen `projectsRoot`. `%APPDATA%\Firepit\settings.json` exists with the seeded value.
- Uninstall via Add/Remove Programs → app + shortcuts gone, settings folder remains (with prompt to remove).
- In a freshly-installed Firepit, opening any workspace tab and invoking `/mcp` inside Claude Code lists `firepit` as a connected server with the expected `firepit_*` tools available. No manual config patching, no PATH editing required. A `Get-Command firepit-mcp` from a freshly-opened PowerShell resolves to the installed binary.
- CI: `firepit-installer.yml` builds the installer on Windows runner, attaches `FirepitSetup.exe` to the GitHub Release alongside the existing publish archive.

## Open questions

- Per-user vs per-machine install — pick per-user as default; offer per-machine via an installer flag for advanced users.
- Code-signing certificate — optional for V1.12; without signing, SmartScreen will warn on first run. Document the workaround in README; revisit when there's a buddy who'll actually run the installer.
- Taskbar pin — Win 10/11 makes pin-by-installer technically possible but unreliable. Skip; document Start-Menu right-click → "Pin to taskbar" instead.

## First-run end-to-end check (template for future planning docs)

After this installer ships, a fresh user's first 5 minutes look like: download → run wizard → pick projects root → land on Firepit → open a workspace tab → see Claude Code start → invoke `/mcp` → see `firepit` listed as connected → call a `firepit_*` tool → it works. Every assertion in that flow must be covered by an acceptance criterion above. Future feature planning docs should include a paragraph in this shape; it forces the spec to think about *resolution / discovery / surface* alongside the artifact being built. (See #6 for the audit that argued for this template.)

## Side note — "where is Claude configured as the AI buddy?"

Today, three places:

1. **Adapter registry** — `MainWindow.xaml.cs` constructs `_adapters` with `ClaudeCodeAdapter` as the only entry. This is the *which* (which adapter handles a project).
2. **`settings.json` → `defaultAgent`** — the *executable name* (default `"claude"`). Resolved via PATH by ConPTY at session start. Override per project with `projects[].agentCommand` / `agentArgs`.
3. **Project markers** — `ClaudeCodeAdapter.Markers = ["CLAUDE.md", ".claude"]`. ProjectDiscovery scans the configured root and qualifies a folder if any registered adapter's markers match.

When a second adapter ships (Aider, Cursor, …) it gets added to (1) — register the adapter, list its markers, and the picker auto-shows folders qualifying for it. (2) and (3) need no change. The "AI buddy" identity is therefore both **adapter-level** (code) and **execution-level** (command name in settings) — kept separate so a project can override which `claude` binary it runs without losing the adapter's behavior contract.
